# B 站写作模式异步创建设计

## 背景

当前 B 站模式创建在一次前台请求中依次完成视频下载、字幕获取、字幕校正和写作模式分析。任务耗时期间页面被阻塞，用户无法先进入模式详情，也无法离开创建页继续操作。

## 目标

- 用户提交 B 站链接后立即获得一个可进入的模式记录。
- 视频获取、字幕识别、AI 字幕校正和写作模式分析全部在后台继续执行。
- 模式列表和详情页能显示处理中、完成、失败三种状态。
- 完成后详情数据自动刷新；失败后保留记录并允许重试。
- 不引入外部队列服务，复用 Studio 当前的后台 Promise、文件状态和 SSE 机制。

## 数据流

1. `POST /api/v1/craft/bilibili/create` 校验 BV 号，生成 craft ID，写入带 `processingStatus: "processing"` 的 `meta.json`，立即返回 craft ID 和占位元数据。
2. 后台任务执行现有的视频/字幕导入、AI 字幕校正、来源文件保存和 `PipelineRunner.analyzeCraft(..., existingCraftId)`。
3. 分析完成后写入正式 `craft_profile.json` 和完成元数据；分析失败则更新元数据为 `processingStatus: "error"` 并保存错误摘要。
4. `GET /api/v1/crafts/:id/status` 返回当前状态、阶段和错误信息。服务重启后，仍可从元数据识别未完成记录，并通过详情页重试。
5. 前端收到立即响应后进入详情页占位态，轮询状态接口；完成后加载正式 profile 和来源文件，失败后显示重试按钮。

## 数据模型

`CraftMeta` 增加可选字段：

- `processingStatus?: "processing" | "ready" | "error"`
- `processingStage?: string`
- `processingError?: string`

旧模式没有这些字段时按 `ready` 处理。处理中记录仍然由模式列表返回，但详情接口不应把“尚未生成 profile”误报成普通不存在。

## 前端行为

- B 站创建按钮只等待快速创建响应，不等待后台解析。
- 详情页显示来源链接、当前阶段、刷新状态按钮；处理中自动轮询，完成后加载完整详情。
- 模式列表卡片显示状态标签。
- 失败状态显示错误摘要和“重新处理”，重试沿用已保存的 BV 号。
- 小说上传继续使用现有同步分析流程。

## 错误和并发

- 同一个 craft ID 同时只允许一个后台任务；重复点击重试时前端按钮禁用，服务端也拒绝重复任务。
- 后台任务任何阶段失败都更新状态，不删除模式记录；临时 source upload 按现有逻辑清理。
- 旧的同步 B 站导入接口保留，避免影响已有调用；新的创建页改用异步接口。
- 状态更新只记录阶段和错误摘要，不把字幕正文写入普通日志。

## 测试

- 创建接口快速返回 processing 记录，并在后台任务未完成时可查询列表和状态。
- 后台成功会使用同一 craft ID 写入 profile 和 ready 元数据。
- 后台失败会写入 error 状态，且重试可再次启动。
- 详情页处理中不要求 profile 存在，完成后能自动切换到正式详情。
- 旧模式元数据无 processing 字段时仍按 ready 展示。
