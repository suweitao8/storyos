# 短片故事候选方案与实时生成设计

## 目标

让短片故事创建先生成一份可审阅、可编辑的完整故事种子，用户确认后再交给当前写作模式生成短片正文；生成期间右侧持续显示已经收到的正文片段和阶段进度，避免长时间只看到转圈。

## 范围

- 只扩展 Studio 的短篇故事创建流程；长篇创建继续使用现有字段和行为。
- 使用已选写作模式中的世界观、故事大纲和写作手法作为参考，生成新的原创故事种子。
- 没有选择写作模式时，仍然允许生成，使用默认短片创作规则。
- 不在候选阶段生成封面或图片；采用并创建后沿用已有故事正文和资产提取流程。

## 用户看到的故事种子

短篇候选包含以下字段，全部在生成完成后可编辑：

1. 故事名称
2. 类型与基调
3. 一句话故事钩子
4. 世界观与运行规则
5. 角色与关系（主角目标、弱点、关键关系）
6. 核心冲突、代价与 stakes
7. 分段故事大纲（开场、升级、转折、高潮、结局）
8. 关键反转与线索回收
9. 结局与情绪余味
10. 画面与声音母题

候选以结构化对象保存于前端状态，同时序列化成已有短篇 `direction` 字符串，保证旧的 `shortRun` action 和下游写作管线仍然能消费完整上下文。用户手动编辑后的字段以最新内容为准。

## 交互流程

1. 用户进入短篇故事创建页并选择写作模式（可选）。
2. 页面自动请求一套候选，也提供“继续随机生成”按钮。
3. 请求期间采用桌面端左右两栏：左侧显示已经完成的候选字段，右侧显示生成阶段、已完成字段、当前模型正文片段和进度；移动端改为上下排列。
4. 右侧只展示正文输出，不展示 thinking/reasoning。按钮在流式生成期间禁用，避免并发候选覆盖。
5. 生成成功后右侧变为“已完成”，左侧所有字段可编辑；用户可以“继续随机生成”或“采用并创建故事”。
6. 点击采用后，把完整结构化候选序列化进短篇创建 action，进入既有短篇生成与故事工作区。
7. 生成失败时保留已收到的正文片段，显示错误和“重试”；不允许把不完整候选直接创建。

## API 与数据流

新增短篇候选流接口，沿用现有 craft profile 加载和故事方向 prompt：

`POST /api/v1/crafts/:id/story-direction/stream`

请求体：`{ kind: "short", language, previousDirection? }`。无写作模式时使用同等能力的通用短篇 prompt（接口使用 `POST /api/v1/story-direction/stream`）。

服务端使用现有 `chatCompletion` 的 `onTextDelta` 回调，将事件写入 SSE：

- `start`: 候选生成开始，包含字段顺序。
- `delta`: 一段最终正文文本；不发送 thinking/reasoning delta。
- `complete`: 完整正文和解析后的 `StorySeed`。
- `error`: 可展示错误信息；如果已有正文，仍保留在右侧。

候选正文使用稳定的 Markdown section 标记，服务端完成时解析为 `StorySeed`；解析缺失必需字段时返回错误而不是生成不可确认的半成品。为了兼容已有生成端点，旧的非流式端点继续返回 `direction`，但短篇创建页改用流接口。

## 思考模式

生成请求显式使用直接输出路径。Provider 层的项目级 `thinkingBudget = 0` 和 `piModel.reasoning = false` 继续作为全局保护；短篇候选新增回归测试，确保客户端请求不会打开 thinking，并且流式 UI 只接收 `delta` 正文。MiniMax 等兼容端点的思考字段仍由 provider 层过滤，不把 reasoning 内容送到 UI 或候选正文。

## 错误处理

- 选定写作模式加载失败：保留通用生成入口，并在页面显示模式加载错误。
- SSE 连接中断：标记候选失败、保留已接收片段、允许重试；不把截断内容发送给创建 action。
- 模型返回无法解析的结构：显示“候选结构不完整”，保留原始片段供排查，允许重新生成。
- 生成按钮、采用按钮在请求期间禁用；旧请求的迟到事件不能覆盖新请求，使用请求序号/AbortController 隔离。

## 测试

- core：短篇候选 prompt 要求十个结构段、原创化约束和不输出思考内容；结构化解析覆盖完整、缺段和 Markdown 代码围栏。
- core/provider：保持 thinking 关闭并过滤 reasoning 内容的回归测试。
- Studio：故事种子序列化、创建 action 传递完整上下文、SSE 事件解析和错误状态测试。
- Studio UI：短篇创建页显示实时面板、生成完成前禁止采用、完成后可编辑并提交。
- 最后运行 core/studio 定向 Vitest、TypeScript 检查和 Studio 构建；如果环境允许，再用 Playwright 验证桌面和移动布局。
