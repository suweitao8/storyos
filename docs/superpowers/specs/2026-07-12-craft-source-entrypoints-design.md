# 新建模式来源入口设计

## 目标

将“新建模式”明确拆成两个来源入口：

1. B 站视频链接：获取视频字幕或音频识别文本。
2. 小说文本文件：上传 `.txt` 或 `.md` 文件并解析章节文本。

两种来源在导入成功后都自动进入同一个写作模式分析接口，并自动保存结果。

## 交互设计

- 新建模式页顶部显示两个来源卡片，分别标注“B 站视频链接”和“小说文本文件”。
- 当前来源卡片保持高亮，页面只显示当前来源对应的输入区，避免 URL、文件结果和错误状态混在一起。
- B 站入口保留现有无 Cookie 字幕策略：优先公开 CC 字幕，没有公开字幕时使用现有音频识别兜底。
- 小说入口保留现有编码检测、章节统计和前 100 章截取逻辑。
- 用户选择通用模式或鬼故事模式后，提交任一来源都会自动执行：导入内容、显示进度、调用 `/craft/analyze`、保存模式并进入模式详情。
- 分析失败时停留在当前来源面板，保留错误信息和最近的进度日志，允许用户重新提交。
- 模式名称继续使用服务端推导结果，沿用去除 URI 编码和末尾章节数量标记的规则。

## 实现边界

- 不新增后端导入接口；复用现有 `/craft/upload`、`/craft/bilibili/import` 和 `/craft/analyze`。
- 在 Studio 前端增加来源类型状态和统一分析请求构造函数。
- 移除手动“提取写作模式”按钮，避免导入与分析分成两个容易遗漏的步骤。
- 通过纯函数测试验证两种来源生成的分析请求都使用正确的文本、名称、语言和模式。

## 验收标准

- 页面同时可见且可选两个来源卡片，切换来源不会显示另一来源的输入控件。
- B 站成功获取字幕后会自动开始模式提取；小说文件上传成功后也会自动开始模式提取。
- 两种路径最终都调用同一分析请求，并在成功后进入模式详情。
- 导入、分析失败时不会卡死按钮，错误和进度可见。
- 现有 B 站解析、小说上传、模式详情及 Studio 构建测试继续通过。
