# 模型配置极简面板设计稿

## 背景

当前 `storyos` 的模型配置界面已经拆到了服务页与详情页，但用户仍然看到过多控制项，和期望的“只保留当前可用模型、API Key、测试连接”不一致。

用户希望的最终状态是：

- 文本、图片、语音三类模型都能配置
- 每个服务只暴露一个当前可用模型
- API Key 默认掩码显示，允许查看、修改
- 支持测试连接
- 保存自动完成，不需要手动点“保存”
- 不需要展示多余的高级项或额外标题
- 旧项目 `animcg` 里已有可复用的 provider 配置和密钥管理模式，尽量借鉴并迁移

## 目标

1. 把模型配置页收敛成一个轻量的服务配置面板。
2. 统一文本、图片、语音三类服务的交互，让它们看起来像同一种卡片。
3. 默认隐藏 API Key，支持掩码查看、编辑、测试连接。
4. 让配置自动保存，避免用户在多个页面重复确认。
5. 尽量复用 `animcg` 的 provider-card 交互和已有配置来源。

## 非目标

- 不增加新的高级配置项
- 不做多模型编排、模型市场或模型排序
- 不做覆盖式的全局配置重构
- 不强制一次性迁移所有历史数据；迁移必须是最佳努力、可回退的

## 方案对比

### 方案 A: 在现有页面上直接补控件

优点：

- 改动最小
- 继续沿用当前的 service detail / service list 页面

缺点：

- 逻辑仍然分散在多个页面和状态层
- 后续再收敛时会继续堆补丁

### 方案 B: 提取统一的 ProviderConfigCard

优点：

- 把“当前模型 + 掩码 Key + 测试连接 + 自动保存”变成单一组件能力
- 文本 / 图片 / 语音可以复用同一套交互
- 更容易把 `animcg` 的 provider-card 模式迁移过来

缺点：

- 需要整理一次当前页面的状态边界

### 方案 C: 重做后台配置模型

优点：

- 最干净

缺点：

- 成本最高
- 对当前目标来说明显过度

## 选择

采用 **方案 B**。

原因很直接：当前问题不是“缺一个功能”，而是“同一类配置在多个地方长得不一样”。先把交互统一，才能把旧项目里的 provider-card 经验平滑迁进来，同时不引入过大的后端重构。

## 设计

### 1. 页面结构

模型配置区域不再显示一个笼统的“模型配置”大标题。改为由三个独立服务卡片组成：

- 文本服务卡
- 图片服务卡
- 语音服务卡

每张卡只展示这几类字段：

- 当前可用模型
- API Key 输入框
- 掩码显示 / 显示切换
- 编辑入口
- 测试连接按钮
- 自动保存状态

如果某个服务当前没有可用配置，卡片仍然保留，但以空状态提示用户补齐。

### 2. 模型选择规则

每个服务只允许一个“当前可用模型”。

实现上保留“模型选择控件”，但行为会被收敛成：

- 只有一个模型时，直接显示该模型并锁定为当前值
- 未来若服务端返回多个候选模型，仍然显示下拉，但默认只会落到当前服务的唯一可用模型

这可以满足当前用户要求，同时保留未来扩展空间。

### 3. API Key 交互

API Key 默认以掩码形式显示。

交互需要支持：

- 默认不明文展示
- 点击“显示”后可查看当前值
- 点击“编辑”后可重新输入
- 输入后自动保存
- 如果后端已经有值，刷新页面后也应以掩码态恢复

UI 行为要和 `animcg` 的 provider-card 一致：用户不需要先删除再重填，也不需要跳到别的页去查密钥状态。

### 4. 测试连接

每张卡都保留独立的测试连接按钮。

测试连接只负责回答两件事：

- 当前 API Key 是否可用
- 当前服务下的当前模型是否可访问

测试结果只在当前卡内展示，不扩散成全局弹窗。

### 5. 自动保存

配置更改后自动保存，不要求用户点击“保存”。

自动保存触发点包括：

- 当前模型变化
- API Key 变化
- 显示状态切换后内容确实改动

保存策略：

- 先做本地去抖
- 再提交到后端
- 成功后刷新卡片状态
- 出错后保留输入内容，并在卡片内展示错误

### 6. 旧项目配置迁移

`animcg` 里已经有 provider 配置、模型默认值和密钥管理模式。这个设计要求实现一个“最佳努力导入”流程：

1. 优先读取当前 `storyos` 自己的配置。
2. 如果当前配置缺失或为空，尝试读取旧项目兼容来源中的 provider 配置。
3. 如果能拿到旧项目中已保存的 API Key，就预填到对应卡片。
4. 如果拿不到，就保持空值，不阻塞页面。

迁移必须满足两个限制：

- 不覆盖用户当前已经改过的新配置
- 不把旧数据当成唯一真相，导入失败时页面仍可正常使用

## 数据流

1. 页面加载时请求当前服务配置。
2. 后端返回：
   - 当前服务
   - 当前模型
   - 可用模型列表
   - API Key 是否已存在
3. 前端把返回结果映射到单一卡片状态。
4. 用户修改字段后，前端在去抖时间内自动保存。
5. 保存成功后刷新卡片状态，并维持掩码显示。
6. 若检测到旧项目兼容配置，先导入到当前运行时，再回填到卡片。

## 错误处理

- 读取失败：卡片级提示，不阻断整个设置页
- 保存失败：保留当前输入，显示明确错误
- 测试连接失败：显示服务端返回的错误摘要
- 旧项目导入失败：静默降级为手动配置，不影响主流程

## 验收标准

1. 设置页不再显示多余的“模型配置”标题。
2. 文本、图片、语音都能看到当前服务对应的唯一可用模型。
3. API Key 默认掩码显示，能切换查看和重新编辑。
4. 可以直接测试连接。
5. 修改后会自动保存。
6. 旧项目的 provider / key 信息若可读，能自动带入；不可读时不影响使用。
7. 页面没有新增多余控制项。

## 需要落地的实现边界

- 前端只负责统一交互和状态承载
- 后端只负责提供当前值、可用模型和密钥读写
- 旧项目迁移优先走兼容读取，不做破坏性写回
