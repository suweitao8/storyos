# Studio 工作区布局扩展设计

## 目标

减少宽屏下页面内容被多层 `max-w-*` 居中容器压缩的问题，让创作相关页面使用更大的可用宽度，同时保留阅读和输入区域的合理行长，并在窄屏下继续自然收缩。

## 现状与问题

- `App.tsx` 对大多数工具页统一使用 `max-w-4xl mx-auto`，页面两侧出现大面积空白。
- 创建故事面板 `StoryCreationPanel.tsx` 继续使用 `max-w-3xl mx-auto`，表单和辅助信息被压成单列窄面板。
- `ChatPage.tsx` 的消息、快捷操作和输入框都各自使用 `max-w-3xl mx-auto`，工作区无法随窗口扩展。
- 写作模式详情虽然已有较宽的外层，但内容层级多，适合改成主内容与辅助信息并列，而不是继续增加卡片内边距。

## 设计方案

### 1. 全局页面容器

在 `App.tsx` 中按页面类型分级：

- 普通设置、数据、导入页面：`w-full max-w-[1440px] mx-auto`，左右使用响应式内边距。
- 写作模式和题材管理：保持宽工作区，使用 `max-w-[1600px]`。
- 聊天、写作、短篇故事：由页面自身占满主区域，不再套普通工具页的居中容器。

页面最大宽度只限制极宽屏的内容密度，不再把 1440px 视口压缩成 896px 的窄列。

### 2. 创建故事布局

`StoryCreationPanel` 改成宽屏双区布局：

- 左侧为主要输入区，承载写作模式、故事方向、标题、类型、字数等核心表单。
- 右侧为创建提示、当前创作约束和提交摘要，在 `lg` 以下回到单列。
- 外层使用 `max-w-[1440px]`，内部网格使用 `minmax(0, 1.35fr) minmax(320px, .65fr)`，避免右栏过窄。
- 文案和控件保持现有行为，不改变提交 payload、流式状态或路由。

### 3. 聊天工作区

聊天页面的历史消息和输入区使用工作区宽度，消息内容本身保留 `max-w-[1100px]` 的可读上限；输入框和快捷操作扩展到同一工作区宽度。这样空白出现在内容两侧而不是整个页面外围，长文本和创建面板不会显得拥挤。

### 4. 响应式与可访问性

- `lg` 断点以下所有双栏回到单列。
- 所有可横向扩展区域使用 `min-w-0`，避免长标题和生成内容撑破页面。
- 不通过隐藏内容解决拥挤问题；内容继续由页面滚动区承载。
- 保持现有按钮、表单标签、焦点样式和键盘顺序。

## 不在本次范围

- 不调整颜色、字体、文案和业务流程。
- 不改变 API、状态模型、路由或数据结构。
- 不把每个页面都改成全宽无边界；详情阅读区域仍保留合理的内容上限。

## 验证标准

- 1440px 宽度下创建故事主区明显扩展，左右辅助区同时可见。
- 1440px 宽度下聊天输入区与消息区不再被 896px 容器限制。
- 375px 和 768px 下无横向溢出，创建故事回到单列。
- Studio 类型检查、相关组件测试和客户端构建通过。
